# DBスキーマ変更履歴（schema-changes/）

## このディレクトリの目的

Phase 2 以降に発生した **DBスキーマ変更を記録する場所** です。
Phase 1 の Gate 5（変更履歴運用）の成果物がここに蓄積されます。

このディレクトリの内容は **クライアント納品物の一部** として扱う。
変更の事実・理由・影響範囲のみを記録し、社内向けの振り返り・教訓は別場所に分ける。

## 記録のタイミング

以下のタイミングで `{連番}-{table}.md` を新規作成：

- Phase 2/3 中にカラム追加・型変更・PK変更・テーブル追加が発生したとき
- Phase 3 から plan-lead がオンデマンド起動された時の対応結果
- マイグレーションファイルを作成する時に、対応する変更記録ファイルも必須

## ファイル命名規則

```
schema-changes/
├── README.md                          ← このファイル
├── 001-courses.md                     ← 連番＋対象テーブル名
├── 002-subscription_plans.md
└── 003-e_learning_purchases.md        ← 既存テーブル拡張も同様
```

連番は3桁ゼロ埋め。新規作成時は最大値+1。

## 社内ナレッジ（振り返り・教訓）の記録場所

★ **このディレクトリには記録しない**。プロジェクト外の社内ナレッジ蓄積場所に保存する：

```
~/.claude/lessons-learned/db-design/
└── portfolio-{連番}-{table}.md
   └── 「なぜ Phase 1 で気付けなかったか」
   └── 「business-interview-helper のテンプレに追加すべき質問」
   └── 「db-design-reviewer のチェック観点に追加すべき項目」
```

## 重要な運用ルール

1. **変更前に必ず plan-lead 経由で承認を得る**
   - dev-mate が直接 schema を変更してはいけない
   - team-lead → plan-lead → db-plan-mate のフローを守る

2. **マイグレーションファイルと変更記録ファイルは1対1**
   - supabase/migrations/ にファイルがあるのに schema-changes/ に記録がない、は禁止

3. **変更ファイル作成と同時に、社内ナレッジ（lessons-learned/）も記録**
   - schema-changes/ は事実のみ（クライアント向け）
   - lessons-learned/ は教訓（Kosuke 社内向け）
   - 両方そろってこそ、同じ失敗を二度しない仕組みになる

4. **連番は飛ばさない**
   - 必ず連続した連番にする（途中削除しない）
